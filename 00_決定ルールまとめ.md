# 決定ルールまとめ

このファイルは、通常チャット運用・Codex運用・保存先運用・案件別ルールの決定事項をまとめた原本です。

## 00. 基本方針

通常チャットを投げ込み口として扱う。

ユーザーが X投稿・スクショ・URL・AI活用ネタ・ツール情報を貼ったら、毎回プロジェクトを開かせたり、モード起動文を貼らせたりしない。こちら側が作業開始前に入力タイプを判定し、保存メモリ内の該当ルールを選んでから進める。

いきなり中身を作らない。まず仕分ける。

## 01. 通常チャットの初動ルール

X投稿・スクショ・URL・AI活用ネタ・ツール紹介が来た場合は、必ず先に以下で返す。

```text
判定：採用 / 一部採用 / 不採用
入れる箱：00_inbox / 01_使える型 / 02_案件別 / 03_Codexに貼るルール / 99_捨てたネタ
使い道：
次の変換先：ChatGPT / Codex / NotebookLM / Canva / 画像生成 / 店舗LINE_FAQ_LP / KDP / 統計12分類 / 個人ストック
今すぐ貼る用：
```

分類後、必要な場合だけ中身・プロンプト・Codex用指示・NotebookLM用指示・Canva用指示に変換する。

## 02. 箱の役割

```text
00_inbox
→ 未整理のX投稿・スクショ・URL・メモ

01_使える型
→ ChatGPTプロンプト、Codexプロンプト、NotebookLMプロンプト、Canva/画像生成の型

02_案件別
→ 店舗LINE_FAQ_LP、KDP_画像生成、統計12分類、個人ストック

03_Codexに貼るルール
→ AGENTS.md、README、命名規則、出力先、Codex用の作業ルール

99_捨てたネタ
→ 不採用・保留・今は使わない情報
```

## 03. ChatGPT側の役割

ChatGPTカスタム指示は「通常チャット用ルールルーター」として使う。

カスタム指示には、作業開始前に入力タイプを判定し、該当する保存メモリ・ユーザー運用ルールを選ぶ指示を入れる。

メモリは補助。通常チャットでの初動ルール、ユーザーが嫌う運用、案件別の注意点を保持する。

通常チャットでは、ユーザーに毎回固定文を貼らせない。

## 04. ai-work-rules リポジトリの役割

`ai-work-rules` は、AI運用ルール・Codex指示・保存先ルールの原本管理用リポジトリ。

制作物本体は置かない。

置くもの：
- AI運用ルール
- Codexに貼るルール
- 保存先ルール
- AGENTS.md追記候補
- README追記候補
- 命名規則
- 決定ルールまとめ

置かないもの：
- 店舗LP本体
- FAQサイト本体
- KDP画像原本
- 生成済みPDF
- 一時ファイル
- 案件ごとの実装ファイル

制作物リポジトリとは混ぜない。

## 05. リポジトリ分離ルール

ルール管理リポジトリと制作物リポジトリは分ける。

```text
ai-work-rules
→ AI運用ルール、Codex指示、保存先ルールだけ

repair-faq-site / lelien-digital-support など
→ 店舗FAQ・LPなど制作物本体

kdp-image-generator
→ KDP画像生成の実装
```

混ぜると、Codexがどの案件の何を直すのか誤認しやすい。保存先、README、AGENTS.md、出力ルールもごちゃつくため、案件ごとに分ける。

## 06. READMEの役割

READMEは人間が最初に見る説明書。

READMEに置くもの：
- このリポジトリの目的
- どのmdを何に使うか
- 更新方法
- 置くもの / 置かないもの

READMEに細かいルール本文を詰め込みすぎない。細かいルール本文は各mdに置く。

## 07. Codex関連ルール

Codex関連では、実装前に必ず「どこに入れるか」を先に判定する。

判定先：
- AGENTS.md
- README.md
- 命名規則
- 出力先ルール
- 個別プロンプト

中身だけ作って終わらない。

Codexに投げる時は、変更対象・禁止事項・変更後の報告形式を明確にする。

## 08. AGENTS.mdに入れるもの

AGENTS.md向き：
- 作業ルール
- 命名規則
- 出力先
- 禁止事項
- ファイル種別の扱い
- Codexが毎回守るべき前提

README向き：
- リポジトリ説明
- 使い方
- 公開手順
- 構成説明

個別プロンプト向き：
- 今回だけの修正
- 1回限りの生成
- 一時的な調査指示

## 09. 保存先ルール

Downloads運用を正にしない。

将来的にはDrive運用へ寄せる。

ただし、こちらで勝手に実際のDriveパスを決め打ちしない。

生成物・PDF・画像・CSV・Markdown・一時ファイルなどの保存先は、個別ファイルにハードコードしない。

`OUTPUT_ROOT` / `EXPORT_ROOT` / `ASSET_ROOT` など、1か所の設定から参照する形にする。

Codexに保存先変更をさせる時は、先に既存の保存先一覧を出させる。

禁止：
- 勝手にファイル削除
- 勝手にフォルダ構成を大幅変更
- 実在しないDriveパスの決め打ち
- 保存先だけでなく中身の仕様まで変更

## 10. ファイル保存・更新の考え方

ローカルやDownloadsに置くだけなら、md更新は手動になりやすい。

GitHubリポジトリに置けば、Codexに更新させられる。

今後ルールを追加する時は、通常チャットで分類し、必要なら `ai-work-rules` の該当mdへCodexまたはGitHub経由で反映する。

## 11. 不確定情報の扱い

ユーザーが「たぶん」「かも」「っぽい」と言った場合、こちらが勝手に前提を確定しない。

不確定情報は不確定のまま扱う。

まず事実だけ整理し、必要な選択肢だけ短く出す。

特にフォルダ整理や保存先では、未確定の場所をこちらで正にしない。

## 12. 質問で止めないルール

曖昧な依頼でも、基本は合理的に仮定して進める。

確認質問は、致命的に必要な時だけ最大1つ。

不明なことは推測で埋めず「不明」と書く。

## 13. 最新情報・規約・料金

最新情報・料金・規約・商用利用・法律・金融・アプリ仕様は、Web検索して確認する。

根拠のない数字、成功率、収益見込みは出さない。

事実・推測・私見を分ける。

断言できない場合は「確度：高/中/低」を付ける。

## 14. 出力スタイル

日本語で短く答える。

基本は関西弁寄り。

前置き・挨拶・褒め言葉はいらない。

同じ内容を繰り返さない。

新情報、具体手順、完成物、コピペ文を優先する。

## 15. LINE / メール返信

LINEやメールの返信文を求められた時は、説明や前置きを付けず、送信本文だけをそのままコピーできる形で出す。

## 16. チラシ・POP・料金表・三つ折り

チラシ・POP・料金表・三つ折りは、完成画像をいきなり作らない。

まずHTML / SVG / PDFで文字編集可能な形を優先する。

画像生成は背景・キャラ・装飾素材だけに使う。

PDFは客に出す納品物。

画像や写真は分析用・元資料として扱い、混同しない。

## 17. PDF / 画像の扱い

PDFは客に出す納品物。

画像・スクショ・撮影ページは、ユーザーの内部分析用・元資料。

統計12分類や資料整理では、PDFを分析元と決めつけない。アップロードされた画像を分析元として扱い、必要な時だけ別途PDFを作る。

## 18. KDP / 画像生成

KDP・画像生成では、サムネではなくPNG原本保存を前提にする。

採用画像だけアップスケールする。

日本語列は管理用。表紙は別扱い。

## 19. 統計12分類の命名規則

統計12分類関連の生成ファイルは、既定の命名規則を守る。

- 英語風・内部ファイル：`12class_`
- 日本語の客向けPDFや納品物：`12分類_`

`12_` や `tokei12_` に勝手に変えない。

## 20. 個人ストック

ユーザーが「個人ストック」と言った場合、店舗案件・修理店・LINE/FAQ/LPの例に寄せない。

個人開発・個人副業・自分用ツール・販売用小アプリなど、店と切り離した視点だけで答える。

## 21. LINE公式URL

LINE公式の友だち追加URLが必要な場合は、既定URLを使う。

```text
https://lin.ee/XrBzzdj
```

新しいURLが提示された場合は、その時点で差し替える。

## 22. X / Twitter URL抽出

X/Twitterリンクだけ投げて解析する運用では、通常チャットで取得できない場合に、手動コピー・全選択・スクショ・テーマだけ代替を同等扱いしない。

通常チャットで取れないなら、要件未達として明確に言う。

必要なら、API・スクレイパー・ブラウザ自動化・Codex側ツールなど、別の自動抽出ルートを提案する。

r.jina.ai等を使う場合も、通常チャットで今できることと外部ツール/設定が必要なことを分けて説明する。

## 23. 通常チャット運用の最終形

ユーザーは普通に貼るだけ。

こちらが勝手に以下を行う。

1. 入力タイプを判定
2. 該当ルールを選択
3. 先に分類を出す
4. 必要なら変換する
5. Codex / NotebookLM / Canva / 画像生成 / 案件別へ流す

ユーザーに毎回モード起動文を貼らせない。

プロジェクトを開く前提にしない。

中身だけ作って終わらない。
